Explanation

The testing process starts by undeploying or deploying the model to get to a known starting state. It will then deploy using instance types specified, one by one, until it finds the first (assuming lowest) instance type that can meet the error rate and response time requirements (see Step 2). Default instance types should cover most cases, but you can override with other types if needed. Default instance count is 1, in order to test how much a single instance can handle. For each instance type, it tests using the TPS rates specified, usually from 1 TPS to 400 TPS, to find how much load a single instance of that type can handle. Default TPS steps should cover most cases, but you can omit or add certain TPS ranges to guide the testing. If an instance type is found, it then extrapolates how many instances of this type would be needed to meet peak TPS based on the load that a single instance was able to handle. To prove that extrapolation, it then runs an endurance test with that instance type and that number of instances, while sending peak TPS (see Step 3). If the endurance test fails, it will try increasing the count and re-testing a few times. The expected result is finding an instance type and count configuration that supports the given requirements. This configuration can then be used either by itself as a Fixed Scale configuration, or as the Maximum Count part of an Auto Scale configuration. For the Minimum Count part of an Auto Scale configuration, check the last section (see Step 4).

The perf test needs exclusive access to the model and its endpoint for the entire duration of the test. The perf test will be deploying and undeploying, and sending various traffic patterns. Do not make your own changes or send your own traffic to the endpoint until the test is over. Do not run tests in parallel for the same model endpoint. Any of these activities may interfere with the test run and invalidate results.
